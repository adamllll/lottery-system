# 奖品图片访问修复记录

## 问题现象

- 创建奖品后，奖品列表中的图片无法显示。
- 服务端日志显示图片请求被 `LoginInterceptor` 拦截：
  - 请求头中的 `user_token` 为 `null`
  - 请求路径形如 `/<uuid>.JPG`
  - 拦截器返回 `401`

## 根因分析

当前图片访问链路依赖两个脆弱前提：

1. 奖品图片入库时只保存文件名，不带固定访问前缀。
2. 前端渲染 `<img src>` 时直接使用数据库中的 `imageUrl` 字段。

这样会导致浏览器直接请求根路径文件，例如：

```text
/9326a174-85ec-48c5-98d5-1c9ca07592e9.JPG
```

而当前登录拦截白名单中只包含：

```text
/*.jpg
/*.png
```

当上传文件后缀为大写 `.JPG` 时，请求不会命中白名单，最终被登录拦截器误判为未登录访问。

## 稳妥修复方案

更稳的长期方案如下：

1. 数据库继续只保存图片文件名，不保存完整 URL。
2. 后端统一将图片对外暴露为固定前缀路径：`/pic/{filename}`。
3. 在 MVC 中显式配置 `/pic/**` 静态资源映射到本地图片目录 `./pic/`。
4. DTO 返回前端时统一把裸文件名转换为 `/pic/...`。
5. 登录拦截器只对白名单路径 `/pic/**` 放行，删除对具体图片后缀的依赖。

这样可以避免：

- `.jpg` / `.JPG` / `.png` / `.webp` 等大小写和格式差异导致的放行失败
- 浏览器图片请求无法携带登录请求头而被误拦截
- 后续切换 OSS/CDN 时前端必须跟着改路径规则

## 当前落地策略

本次先采用最小修复方案恢复功能：

- 在拦截白名单中补充大写图片后缀，例如 `/*.JPG`、`/*.PNG`

原因：

- 改动最小，风险最低
- 可以立刻恢复当前奖品图片显示
- 不影响已有数据库数据和前端逻辑

## 后续建议

后续如果继续演进图片模块，建议按“稳妥修复方案”整体落地，把图片公开访问从“按文件后缀猜测是否放行”升级为“按固定资源路径统一管理”。
